# Section 7.C / 7.D - Blockchain Performance and Deployment

## 7.C Blockchain Performance Analysis

In this section, we evaluate the blockchain-layer performance of SCAPE-ZK using the deployed Hyperledger Fabric test network. Following the performance-analysis style used by SSL-XIoMT, the experiment reports the measured runtime behavior of each implemented operation and then explains the graph trend rather than introducing synthetic values. The benchmark focuses on the ledger-facing part of SCAPE-ZK: registering encrypted EHR metadata, recording proof-verification outcomes, revoking users, updating credential state, and querying whether a record exists.

The SCAPE-ZK smart contract was implemented in Go using the Hyperledger Fabric Contract API for Go. The benchmark workload was written in JavaScript for Hyperledger Caliper and executed through Node.js. The result-processing scripts used Python; the locked throughput figure was generated from the Caliper summary CSV using Matplotlib, while auxiliary SVG summaries were produced from the same CSV using Python standard libraries such as `csv`, `pathlib`, and XML escaping utilities. No baseline TPS values were fabricated: all throughput and latency values reported in this subsection are taken from the ten logged Caliper trials in the repeated benchmark result set.

Each operation was evaluated at 100, 500, and 1000 submitted transactions. One warm-up run was excluded from the reported statistics, after which the full benchmark was repeated ten times. The reported values are the mean and sample standard deviation across those ten trials. Across all operation/load combinations, the benchmark completed with zero failed transactions.

The locked Section 7.C graph is `fig_fabric_tps_vs_load.{png,pdf}`. It was generated from the ten-trial Hyperledger Caliper summary and includes the four ledger write operations: `Register`, `VerifyProof`, `Revoke`, and `UpdateCred`.

As shown in the graph, the four write operations follow a similar scaling pattern as the submitted load increases. At 100 transactions, the write throughput is approximately 19-20 TPS: `Register` reaches 19.34 +/- 3.26 TPS, `VerifyProof` reaches 20.07 +/- 3.30 TPS, `Revoke` reaches 19.87 +/- 3.03 TPS, and `UpdateCred` reaches 20.28 +/- 1.49 TPS. At 500 transactions, the same operations increase to approximately 131 TPS, with `Register` at 131.33 +/- 2.51 TPS, `VerifyProof` at 130.99 +/- 4.93 TPS, `Revoke` at 130.83 +/- 4.76 TPS, and `UpdateCred` at 131.65 +/- 5.06 TPS. At 1000 transactions, the write operations remain closely grouped: `Register` reaches 215.15 +/- 14.24 TPS, `VerifyProof` reaches 221.19 +/- 8.03 TPS, `Revoke` reaches 221.20 +/- 13.61 TPS, and `UpdateCred` reaches 221.56 +/- 10.47 TPS.

This graph indicates that the measured Fabric deployment scales upward over the tested load range without shifting the bottleneck to one specific SCAPE-ZK write function. The four write functions have similar throughput because they all follow the same Fabric transaction path: proposal execution, endorsement, ordering, validation, commit, and world-state update. Their chaincode logic writes compact metadata or status records, so the measured difference among `Register`, `VerifyProof`, `Revoke`, and `UpdateCred` is small compared with the common cost of the Fabric transaction pipeline.

The read-only `RecordExists` query is not included in the write-operation graph because it does not follow the same write-transaction path. Its measured throughput is higher: 112.98 +/- 1.41 TPS at 100 transactions, 447.05 +/- 15.10 TPS at 500 transactions, and 702.38 +/- 38.86 TPS at 1000 transactions. This result is expected because `RecordExists` avoids ordering, validation, commit, and world-state update overhead. The separation between read and write performance supports the intended SCAPE-ZK design, where expensive cryptographic work is performed off-chain and the blockchain stores compact authorization, revocation, and audit state.

Average latency also remains low at the 1000-transaction load. `Register` records 89 +/- 12 ms, `VerifyProof` records 87 +/- 7 ms, `Revoke` records 86 +/- 8 ms, and `UpdateCred` records 92 +/- 8 ms. `RecordExists` completes in 10 +/- 0 ms on average. These values show that the tested Fabric deployment can sustain hundreds of SCAPE-ZK ledger write operations per second while keeping average write latency below 100 ms at the highest tested load.

The `VerifyProof` result should be interpreted carefully. In the current prototype, the Fabric chaincode records a proof-verification or aggregate-authorization result as ledger state; it does not execute full Groth16 proof verification or BLS aggregate-signature verification inside chaincode. Therefore, the `VerifyProof` throughput and latency reported here measure the blockchain transaction cost of anchoring the verification outcome. End-to-end authorization latency must additionally include the off-chain cryptographic operations evaluated in the computation-cost analysis.

## 7.D Blockchain Deployment

SCAPE-ZK was deployed on Hyperledger Fabric because the target healthcare scenario is a permissioned consortium rather than an anonymous public-chain environment. Hospitals, clinics, laboratories, insurers, and regulators can be represented as known consortium participants with managed identities. Fabric provides membership management, endorsement policies, channel-based isolation, deterministic chaincode execution, and an auditable world state, which match the trust and compliance requirements of cross-domain EHR sharing.

The deployed SCAPE-ZK chaincode exposes five benchmarked functions. `Register` anchors encrypted EHR metadata, including the content identifier, Merkle root, binding tag, policy digest, owner, and transaction timestamp. `VerifyProof` records an approved proof or aggregate-authorization context using the context identifier, batch identifier, and aggregate signature field. `Revoke` updates revocation state, `UpdateCred` changes credential state, and `RecordExists` checks whether a ledger key is present. The implementation is intentionally compact: it stores and updates authorization state but does not place large EHR files or expensive cryptographic proving work on-chain.

This deployment follows the SCAPE-ZK system division. Zero-knowledge proof generation, CP-ABE encryption, proxy re-encryption, Merkle proof construction, and aggregate-authentication processing are handled by off-chain clients or service components. Fabric is used to preserve the shared state that must be consistent across domains, including registration records, authorization outcomes, revocation visibility, credential updates, and audit evidence. This division reduces chaincode computation while keeping the blockchain useful for tamper-evident coordination among healthcare domains.

The deployment results complement the analytical scalability comparison, but they are not a direct TPS comparison against XAuth, SSL-XIoMT, or the IIoT SSI identity-resolution scheme. A direct numeric TPS comparison would require implementing each baseline in the same Fabric/Caliper environment under the same endorsement policy, transaction mix, hardware, and load profile. Since those full baseline deployments were not implemented in this workspace, this paper does not report fabricated baseline blockchain throughput. The contribution of the deployment experiment is instead to show that the SCAPE-ZK blockchain interface runs repeatedly on a realistic permissioned ledger stack with zero failed transactions in the tested workload.

Overall, the blockchain experiment supports the practical deployment claim of SCAPE-ZK. The system keeps heavyweight cryptographic operations off-chain while using a consortium ledger for compact authorization state, revocation, consistency, and auditability. The measured write-operation range of approximately 215-222 TPS at 1000 submitted transactions, together with sub-100 ms average write latency, indicates that the implemented blockchain layer is practical for anchoring SCAPE-ZK state transitions in the tested local Fabric environment.
